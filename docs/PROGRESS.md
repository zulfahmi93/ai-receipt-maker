11. **Rounding mode = `MidpointRounding.AwayFromZero`** (not `ToEven`). Decided 2026-05-07 at start of Phase 2c, supersedes plan T2b.10's original `ToEven` wording. Banker's rounding (`ToEven`) is an IEEE 754 / accounting-statistics convention that minimizes bias when many values are summed in books — it is **not** the consumer-receipt norm. Real-world POS, tax authorities, and shoppers expect round-half-up: Japan 四捨五入 (¥12.5 → ¥13), Malaysia LHDN guidance, US sales-tax practice, etc. `ToEven` would surprise users at every `.x5` midpoint (¥12.5 → ¥12, $1.245 → $1.24). The flip applies to both `ReceiptCalculator` (one line: `Math.Round(value, dp, AwayFromZero)`) and the upcoming `MoneyFormatter` so the displayed total can never disagree with the calculated total at a midpoint. T2b.10 theory data updated: `(12.45 → 1.25, 12.55 → 1.26)`. Plan file annotated; T2c.2 expectation `JPY 12.5 → ¥13` now matches the calculator.
12. **T2c.5 time-format spec corrected.** Plan originally wrote `hh:mm a` → `10:42 AM`. Two issues found at Phase 2c start (2026-05-07): (a) `.NET DateTime.ToString` does not recognize a lone `a` as the AM/PM designator — the specifier is `tt`. Lone `a` is emitted literally (probe: `"10:42 a"`). (b) The plan implied the same `ms-MY` plumbing as T2c.4, but `ms-MY`'s AM designator is `"PG"` (Pagi), not `"AM"`. Resolution: T2c.5 fixture uses `TimeFormat="hh:mm tt"` and `Locale=null` (invariant fallback) to produce `"10:42 AM"`. The formatter still routes through `options.Locale → CultureInfo.GetCultureInfo` with invariant fallback — same plumbing as T2c.4, just exercised with a different locale value.

13. **`TextMeasurer.WrapLines` does NOT break single oversize words at glyph boundary.** Decided 2026-05-07 at Phase 3 close. The original plan asked for both (a) glyph-boundary breaking when one word exceeds `maxWidth` AND (b) a `string.Join(" ", lines) == originalText` round-trip invariant (T3.8). The two are mutually exclusive — splitting a word into multiple line-segments and rejoining with spaces re-inserts a space that wasn't in the source. Kept the round-trip invariant (test T3.8 holds); oversize words are placed alone on their own line and accept visual overflow. XML doc on `WrapLines` documents this. Phase 3b's section renderers can revisit if a real fixture demonstrates a pathological long-word case (e.g. an SKU/URL longer than column width); fix at the renderer layer (truncate with ellipsis, or different break rule for known long-word columns) rather than mutating WrapLines into a stateful "split mode" toggle.

14. **T3.5 FamilyName assertion relaxed from `==` to `Contains` (case-insensitive).** Decided 2026-05-07 at Phase 3 close. SkiaSharp 3.119.2 reports the Inter variable font's `FamilyName` as `"Inter Variable"` (matches the VF's typographic family record), not the bare `"Inter"` the original RED spec assumed. Static Inter cuts report `"Inter"`. The test's intent is "embedded font loaded, not a system fallback" — `Contains("Inter")` proves that for both VF and any future static-cut fallback without re-flapping if SkiaSharp's name-table mapping ever changes. Inter VF (rsms/inter v4.1, divergence #2 / ADR 0004) remains the embedded asset; no font swap.

17. **CustomerCashierSection — fixed `ColumnGutter = 16f` between left and right columns.** Decided 2026-05-08 at Phase 3b V9.3 sign-off (UI/UX Expert blocker B1). Original implementation split the section width 50/50 with `LeftColumnFraction = 0.5f`; with a long left-column value (e.g. `customerName="Walk-in Customer"`) the value's right-edge abutted the right column's label left-edge at the midpoint, so PdfPig text extraction read the row as `"Walk-in CustomerCashier"` and the rendered PNG showed the two columns visually touching. Fix: replaced the fraction with a fixed 16px gutter; both columns now share `(width - ColumnGutter) / 2`. Regression guard is **pixel-mode** rather than PDF text — PdfPig's `Page.Text` strips horizontal whitespace within a row and concatenates adjacent draws regardless of the on-canvas gap, so a substring assertion would never fail. The new `CustomerCashierSection_LeavesGutter_BetweenColumns` renders the section onto an `SKBitmap` and asserts a 5-column-wide vertical strip centred on midX is paper-coloured.

18. **FooterSection — body and contact lines now wrap via `TextMeasurer.WrapLines`.** Decided 2026-05-08 at Phase 3b V9.3 sign-off (UI/UX Expert blocker B2). Original implementation drew each body string as a single un-wrapped row, so the sample fixture's legal note (`"This receipt is computer generated and does not require a signature."`) overflowed past the canvas right edge as `"…does not require a signa"`. Refactor introduced a `private readonly record struct LineSpec(string Text, SKFontStyleWeight Weight, float FontSize)` and unified body + contact rendering through `MaterializeBodyEntries` / `MaterializeContactEntries`. `Measure` sums `WrapLines(...).Count` across all entries; `Draw` paints each wrapped line at its own typeface + size. New regression test `FooterSection_WrapsLongBodyLines_AtAvailableWidth` asserts `Measure(long) > Measure(short)` at narrow width and uses **per-word presence** rather than full-string substring — PdfPig drops whitespace at line-wrap boundaries (concatenates `"note"` and `"used"` across rows into `"noteused"`), so a verbatim contains check on the source line is unreliable; per-word is robust to that quirk. The existing T3b.20 verbatim assertion at default width=360 still passes because the sample legal note happens to fit on one line at 360px width — under the new wrap path, only narrower widths trigger the wrap.

Plan line 532 reads `Changing theme.accentColor to red changes TOTAL bar pixel sample.` Codebase truth (shipped in Phase 3b sub-cluster C, T3b.11–T3b.17): `TotalsSection.Draw` binds the TOTAL bar fill to `data.Theme?.HighlightColor` → `ThemeColors.DefaultHighlightColor` (LightGray fallback). `QrSection.Draw` binds QR modules to `data.Theme?.AccentColor` → `ThemeColors.DefaultAccentColor` (DarkSlateGray). Resolution: T3c.3 test mutates `HighlightColor` and leaves `AccentColor` untouched so QrSection is unaffected. Plan wording is stale; codebase wins.

19. **MetaSection — combined "Date & Time" row when both formats produce output.** Decided 2026-05-08 at Phase 3b V9.3 sign-off (UI/UX Expert blocker B3). Mockup pairs the two values into a single row labelled `"DATE & TIME"`; original implementation emitted two separate `Date` / `Time` rows. Combined-row test asserts both label substring and exact joined value substring in PdfPig text plus a height-equality check against `dateFormat`-only fixture (proves the combined row is exactly one row, not two).

21. **T3c.8 (`layout.showShadow`) deferred to Phase 3d.** ~~Decided 2026-05-08 at Phase 3c sub-cluster C open.~~ Resolved 2026-05-08 at Phase 3d C-B close: T3c.8 landed via `RenderContext.EmitShadow { get; init; }` (default false). Plan line 537 reads "Changing `layout.showShadow=true` draws shadow on PNG only (PDF unaffected)." `SkiaReceiptRenderer.Render` operates against an `SKCanvas` and is canvas-agnostic by design (one render function drives PDF + PNG + SVG via three different `SKCanvas` sources — see ADR 0001). Shadow-on-PNG-only required the renderer to know its output target, which would have violated that contract. Resolution: `EmitShadow` flag added to `RenderContext`, honored by `SkiaReceiptRenderer.Measure` (canvas grows by `ShadowOffset=8f` on right + bottom edges) and `SkiaReceiptRenderer.Render` (paints `SKColors.Black.WithAlpha(64)` shadow rect via static `DrawShadow` helper before paper paint, mirrors paper's RoundRect path when `BorderRadius>0`, AA on for soft edges vs. paper's AA off). `PngExporter` ctor `emitShadow=true` default — opt-out via `new PngExporter(fonts, scale: 2, emitShadow: false)` (T3d.6 uses this for clean `width == receiptWidth × scale` assertion). `PdfExporter`/`SvgExporter` leave `EmitShadow` at default false — vector consumers want flush edges. Test pinned at renderer level (`SkiaReceiptRendererEmitShadowTests.cs`, single `[Fact]` asserting Measure dims grow + non-zero alpha shadow pixel beyond paper rect) — keeps the contract owner (composer) authoritative; raster exporters merely opt in. ShadowOffset chosen at 8f (not 4f) so AA-feathering at shadow edges doesn't endanger the test's 1-pixel-past-paper sample point.

24. **CLI process tests resolve the CLI binary via `AssemblyMetadata`, not a hardcoded path.** Decided 2026-05-08 at Phase 4 C-A open. Phase 4 ships a CLI binary (`receipt-toolkit.dll`); the test contract is the **shipped process boundary** (parser config + exit codes + stderr formatting all live in `Program.cs` + `System.CommandLine`'s setup), so in-proc `Main(args)` invocation would bypass the parser and validate nothing. `Process.Start("dotnet", "exec", "<path>/receipt-toolkit.dll", …)` is the only honest test surface. The path must survive Debug/Release config + path-prefix differences across dev boxes. Resolution: `tests/ReceiptToolkit.Cli.Tests/ReceiptToolkit.Cli.Tests.csproj` declares `<AssemblyMetadata Include="ReceiptToolkit.Cli.AssemblyPath" Value="$(MSBuildThisFileDirectory)..\..\src\ReceiptToolkit.Cli\bin\$(Configuration)\net10.0\receipt-toolkit.dll" />`. `CliRunner` reads the value via `Assembly.GetCustomAttributes<AssemblyMetadataAttribute>()`, normalises with `Path.GetFullPath`, and throws `FileNotFoundException` with a "run dotnet build first" hint when missing. Net effect: zero hardcoded paths in test source, no environment variables, no per-run `--cwd` dance, and the wiring lives in the csproj where reviewers expect to find build configuration. Phase 5 (API integration tests) and Phase 6 (Telegram bot worker tests) can re-use the same pattern when they need to spawn their own hosts.

22. **`PdfExporter.DefaultPageHeight` 1024 → 1200.** Decided 2026-05-08 at Phase 3d C-A GREEN. RED-phase Test Engineer used `DefaultPageHeight=1024` for both halves of T3d.4 (sample fixture must produce 1 page; long-items fixture must produce ≥2). GREEN-phase .NET Expert ran `SkiaReceiptRenderer.Measure` against both fixtures and surfaced the actual composition heights: sample = **1040px** (40px over 1024), long-items = **2010px**. At `PageHeight=1024`: `Math.Ceiling(1040/1024) = 2 pages` for sample — contradicts the test assertion. Plan T3d.4 itself only specifies the *behavioral invariant* ("sample → 1 page; long → ≥2"), never names a page-height constant. The 1024 value was an arbitrary scaffold default in the orchestrator's RED prompt — XML doc claimed "≈A4 portrait at 360px width" but A4 at 360px-wide ratio is ~509px, not 1024. Resolution: bump `DefaultPageHeight` to 1200 with XML doc updated to cite "sample composes to ~1040px" as the rationale. Long-items fixture (2010px) still produces `ceil(2010/1200)=2` pages — invariant preserved. Per "no silent compromises" feedback rule, the bump was surfaced explicitly rather than fudging the ceil formula or rewriting the test post-RED. **Lesson for future clusters:** when a test couples to a numerical constant interacting with measured behavior, run `Measure(...)` *before* the RED prompt — or pin the *behavioral invariant* not the *constant value* (e.g. have the test pick a per-fixture pageHeight rather than coupling to the production default).

25. **`ReceiptValidator` DI registration must use an explicit factory, not `AddSingleton<ReceiptValidator>()`.** Discovered 2026-05-08 at Phase 5 sub-cluster C-A GREEN. `ReceiptValidator` exposes two public constructors: `()` (binds `DefaultRules()` — 12 rules) and `(IEnumerable<IValidationRule>)` (binds whatever the caller supplies). The Microsoft.Extensions.DependencyInjection container picks the constructor with the most parameters it can resolve, and `IEnumerable<IValidationRule>` resolves to the empty enumerable when no `IValidationRule` services are registered — there is no service to fail-fast on. Result: with `services.AddSingleton<ReceiptValidator>()` the container instantiated a validator with **zero rules**, so every `POST /api/receipts/validate` call returned `valid:true` regardless of input (debug probe surfaced a fresh `new ReceiptValidator()` returning 1 error vs. the injected instance returning 0). Fix: register via factory `services.AddSingleton(_ => new ReceiptValidator())` so the parameterless constructor runs explicitly. Same pattern needed for any future Core type with a multi-ctor "DI overload" — Phase 6 (Telegram bot) and Phase 7 will likely re-encounter it. **Alternative considered, rejected:** mark the `(IEnumerable<IValidationRule>)` ctor `[ActivatorUtilitiesConstructor]`-free and instead register each `IValidationRule` as its own service. This would let the container fill the enumerable with all 12 rules, but it pulls validation-rule wiring up into every consumer's DI container (CLI, API, bot) instead of staying encapsulated inside `ReceiptValidator.DefaultRules()`. Factory keeps the single-source-of-truth in Core. **Test that exercised the bug:** `ValidateEndpointTests.Post_Validate_InvalidData_Returns200WithErrors` — expected `valid:false` + `errors[]` carrying a `business.businessName` field, observed `valid:true` + empty errors before the factory fix.

26. **Bot namespace `ReceiptToolkit.TelegramBot.Telegram` renamed → `ReceiptToolkit.TelegramBot.Messaging`.** Decided 2026-05-08 at Phase 6 sub-cluster A. Original layout placed `ITelegramSender` under `ReceiptToolkit.TelegramBot.Telegram` (matching the file's "Telegram" responsibility) but `Telegram.Bot.Types` (the upstream NuGet package) cannot be imported alongside it without ambiguity — `using ReceiptToolkit.TelegramBot.Telegram` shadows the root `Telegram` namespace and `Telegram.Bot.Types` resolution fails (`error CS0234: type 'Bot' does not exist in namespace 'ReceiptToolkit.TelegramBot.Telegram'`). Fix is one-line per file, but the underlying lesson is: **never name a project sub-namespace after a root namespace from a referenced NuGet package**. Renamed to `Messaging` (semantically accurate — `ITelegramSender` is the messaging seam). All handler + DI usings updated; no public-API surface change for downstream consumers because Phase 6 is the first to ship the bot's public types.
